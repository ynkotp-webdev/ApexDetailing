# Apex Detailing — сайт-портфоліо детейлінг-студії

Багатосторінковий маркетинговий сайт для вигаданої детейлінг-студії у Львові, побудований на Astro з TypeScript та Tailwind CSS v4. Проєкт демонструє повний цикл розробки промо-сайту для локального сервісного бізнесу: від семантичної верстки й дизайн-системи до SEO, доступності та автоматичного деплою.

---

## Опис

Це навчальний проєкт, створений для портфоліо. Бренд **Apex Detailing**, адреса, телефони та email — вигадані; будь-який збіг із реальними компаніями випадковий (це прямо зазначено і в футері самого сайту).

Мета проєкту — показати, як я підходжу до розробки типового комерційного сайту-візитки для сервісного бізнесу: структура сторінок, що конвертує відвідувача в заявку, продумана навігація, форма зворотного зв'язку з валідацією, коректний SEO-каркас і адаптивна верстка під усі пристрої.

Сайт статично згенерований (SSG на Astro) і розгортається на GitHub Pages через автоматичний CI/CD-пайплайн.

## Основні можливості

- **Адаптивна навігація** — окремі компоненти для десктопного горизонтального меню (`Desktop.Nav.astro`) і мобільної бургер-панелі (`BurgerButton.astro` + `BurgerPanel.astro`), що керується vanilla JS (`menu.js`): відкриття/закриття, оверлей, клавіша Esc, закриття по кліку на пункт меню, блокування скролу `body` під час відкритої панелі.
- **Форма зворотного зв'язку з клієнтською валідацією** (`Form.astro`) — перевірка імені, телефону у форматі `+380XXXXXXXXX` (regex), опису авто та обов'язкового вибору послуги; помилки показуються inline при `blur`/`input`, статус відправки виводиться в `aria-live="polite"`-регіон, сама відправка йде асинхронно через `fetch` на сторонній сервіс [Formspree](https://formspree.io/).
- **Прайс-лист і FAQ на нативних акордеонах** — картки послуг (`CardItem.astro`) та питання (`FAQ.astro`) реалізовані через семантичні `<details>/<summary>`, а не кастомний JS-акордеон: доступність і клавіатурна навігація працюють без жодного скрипта.
- **Оптимізовані зображення** — усі фото на сторінках проходять через `<Picture>`/`<Image>` з Astro: автогенерація `avif`/`webp`, набір `widths` під різні viewport, `loading="eager"`/`fetchpriority="high"` для контенту над згином і `loading="lazy"` для решти.
- **SEO-каркас** — унікальні `title`/`description`/`og:image` для кожної сторінки (проброшені як `Props` у `Layout.astro`), канонічні URL, Open Graph та Twitter Card метатеги, автоматична `sitemap.xml` (`@astrojs/sitemap`) і динамічно згенерований `robots.txt` (`src/pages/robots.txt.ts`), що посилається на sitemap.
- **Самохостинг шрифтів** — `Inter` та `Playfair Display` (кирилиця + латиниця, потрібні накреслення) підключені локально через `@fontsource`, без запиту до Google Fonts — коментар у коді explicit пояснює, що це зроблено для усунення render-blocking запиту.
- **Вбудована Google Maps карта** з `loading="lazy"` на сторінці контактів.
- **Кастомна сторінка 404** з логотипом і кнопкою повернення на головну.
- **Автоматичний CI/CD** — GitHub Actions workflow (`.github/workflows/deploy.yml`) білдить і деплоїть сайт на GitHub Pages при кожному пуші в `main`.

## Структура сайту

| Сторінка | Маршрут | Призначення |
|---|---|---|
| **Головна** | `/` | Hero-екран з CTA, блок переваг («Чому обирають нас»), секція «Про нас» (`#about`) з філософією студії |
| **Послуги та ціни** | `/services` | Вступний блок, прайс-лист з 5 послугами у вигляді акордеонів (`#prices`, детальний перелік робіт і ціна для кожної), блок FAQ |
| **Контакти** | `/contacts` | Інформаційний блок з CTA, карта Google Maps + адреса/телефони/email/графік роботи, форма заявки (`#form`) |
| **404** | `/404` | Кастомна сторінка «не знайдено» з логотипом і посиланням на головну |

Окремої сторінки «Ціни» немає — прайс-лист реалізований як секція всередині сторінки «Послуги» (це відображено чесно, без домислювання неіснуючого маршруту).

## Технології

Список сформований виключно на основі `package.json`, `astro.config.mjs` та фактичного коду проєкту:

- **[Astro 6](https://astro.build/)** — статична генерація сайту (SSG)
- **TypeScript** — `tsconfig.json` розширює `astro/tsconfigs/strict`; типізація використана в пропсах компонентів (`CardItem.astro`, `CPAButton.astro`) і в клієнтському скрипті форми (`Form.astro`)
- **[Tailwind CSS v4](https://tailwindcss.com/)** — підключений через `@tailwindcss/vite`, кастомна дизайн-система у `@theme`
- **PostCSS** — `@tailwindcss/postcss` у білд-пайплайні
- **[@astrojs/sitemap](https://docs.astro.build/en/guides/integrations-guide/sitemap/)** — автоматична генерація sitemap
- **[@fontsource/inter](https://fontsource.org/) та @fontsource/playfair-display** — самохостинг шрифтів
- **Vanilla JavaScript** — логіка бургер-меню (`src/scripts/menu.js`)
- **Formspree** — сторонній бекенд для прийому даних з форми (без власного сервера)
- **GitHub Actions** (`withastro/action`, `actions/deploy-pages`) — CI/CD деплой на GitHub Pages

## Особливості реалізації

- **Компонентна архітектура за доменами.** Компоненти згруповані не «за типом», а за зоною відповідальності: `LayoutComponents/` (header/footer), `Menu/` (навігація), `indexComponents/`, `ServicesComponents/`, `ContactsComponents/`, окремий `data/nav.ts` для єдиного джерела правди по пунктах меню (використовується і в десктопній навігації, і в мобільній панелі, і у футері).
- **Дизайн-система на Tailwind v4 `@theme`.** У `global.css` задані кастомні CSS-змінні палітри (`cobalt`, `carbon`, `clinic`) і шрифтових токенів, а типові UI-патерни винесені в перевикористовувані класи-компоненти (`.btn-primary/outline/ghost`, `.field`, `.field-error`, `.section-title`, `.section-eyebrow`, `.container-x`) через `@layer components` — замість дублювання одних і тих самих utility-класів по всіх сторінках.
- **Дані відокремлені від розмітки.** Прайс-лист (`Cards.astro`), переваги (`WhyOurServices.astro`) та FAQ (`FAQ.astro`) описані як масиви даних і рендеряться циклом через один компонент-картку (`CardItem.astro`, `WhyOurServicesItem.astro`) — додати новий пункт означає додати один об'єкт у масив, а не копіювати блок розмітки.
- **Доступність (accessibility) продумана на рівні розмітки, а не додана «зверху».** `aria-expanded`/`aria-controls`/`aria-hidden` на бургер-меню, `aria-live="polite"` на статусі форми, `aria-invalid` + прив'язані повідомлення про помилки (`data-error-for`) на полях форми, `aria-label` на іконкових посиланнях (соцмережі, логотип), фокус-стилі (`focus-visible:ring`) на інтерактивних елементах, семантичні `<header>/<nav>/<main>/<footer>/<section>/<article>` замість `div`-супу, нативні `<details>/<summary>` для акордеонів.
- **Продумана стратегія завантаження зображень.** `<Picture>` генерує `avif`/`webp` з набором `widths`, а `sizes` підібраний під реальну поведінку компонента (`100vw` для повноекранних hero-банерів, `(min-width: 768px) 50vw, 100vw` для карток у сітці); hero-зображення на головній вантажиться з `fetchpriority="high"`, решта — `lazy`.
- **Прив'язка якоря під sticky-хедер.** Правило `:target { scroll-margin-top: 6rem }` у `global.css` компенсує прилиплий header, щоб переходи по `#about`/`#form` не ховали заголовок секції під хедером.
- **Клієнтська валідація форми написана вручну, не «навколо» браузерного `required`.** Кожне поле має окрему функцію валідації, помилки виводяться біля конкретного поля, стан `aria-invalid` синхронізований з візуальним станом, а сабміт іде через `fetch` з обробкою мережевої помилки та неуспішної відповіді сервера окремо.

## Адаптивність

Верстка побудована mobile-first на утилітах Tailwind (`sm:`, `md:`, `lg:`):

- Навігація: на мобільних — бургер-меню з висувною панеллю на весь екран, на `lg:` — горизонтальне меню в хедері (обидва компоненти в DOM, видимість перемикається CSS-класами `lg:hidden`/`hidden lg:flex`, без дублювання JS-логіки для двох станів).
- Сітки контенту: блок переваг (`WhyOurServices`) — 1 колонка на мобільному, 2 на `md:`; FAQ — 1 колонка на мобільному, 2 на `sm:`; блок «Про нас» і контакти на десктопі — `lg:grid-cols-2`, на мобільному — вертикальний стек з `order-*` для контролю порядку блоків.
- Зображення: параметр `sizes` у `<Picture>` окремо підібраний під layout кожного блока, тож браузер завантажує зображення потрібного розміру, а не найбільше доступне.
- Форма: поля йдуть в одну колонку на мобільному і `sm:grid-cols-2` на ширших екранах.
- Прайс-картки (`CardItem`) на мобільному — фото зверху, текст знизу; на `sm:` — горизонтальний ряд з фото і контентом поруч.

## Сильні сторони проєкту

Кожен пункт нижче підтверджується конкретним кодом у проєкті, а не є загальною фразою:

1. **Реальна дизайн-система, а не набір випадкових utility-класів.** Кольори, шрифти й повторювані UI-патерни винесені в `@theme` і `@layer components` — зміна фірмового кольору чи стилю кнопки робиться в одному місці.
2. **Доступність вбудована в розмітку з самого початку**, а не імітована: клавіатурна навігація по бургер-меню, коректні ARIA-атрибути, зв'язок помилок форми з полями, семантичні теги.
3. **SEO зроблено системно, а не для однієї сторінки.** Кожна сторінка передає власні `title`/`description`/`image` в `Layout`, є canonical URL, OG/Twitter метадані, sitemap і `robots.txt`, що на нього посилається.
4. **Продумана продуктивність завантаження**: сучасні формати зображень (`avif`/`webp`) з responsive `srcset`, самохостинг шрифтів замість зовнішнього запиту до Google Fonts, `lazy`/`eager` завантаження там, де це доречно.
5. **Форма зворотного зв'язку зроблена як у продакшн-проєкті**, а не як демо: валідація по кожному полю окремо, доступні повідомлення про помилки, обробка мережевих і серверних помилок, відсутність перезавантаження сторінки.
6. **Чиста, передбачувана структура компонентів**, згрупована за доменами сторінки, з єдиним джерелом даних для навігації — легко масштабувати на нові сторінки чи секції.
7. **Готовий CI/CD.** Проєкт не просто «збирається локально» — є робочий автоматичний деплой на GitHub Pages при пуші в `main`.

## Що можна покращити

Чесна оцінка того, що варто доопрацювати в наступних ітераціях:

- **Немає автоматизованих тестів** (unit/e2e) — якість наразі підтверджується лише ручною перевіркою та типізацією.
- **CI не перевіряє якість коду перед деплоєм** — workflow виконує лише `build`, без кроків `astro check`/лінтингу, тож помилка типів чи лінта не заблокує деплой.
- **Валідація телефону жорстко прив'язана до українського формату** (`+380XXXXXXXXX`) — для міжнародного бізнесу знадобиться гнучкіший підхід.
- **Немає структурованих даних Schema.org** (наприклад, `LocalBusiness`) — для локального сервісного бізнесу це дало б додаткову перевагу в пошуковій видачі.
- **`package.json` зберігає технічну назву `repository-template`** — косметичний момент, який варто привести у відповідність до назви проєкту.

## Встановлення та запуск

Команди відповідають скриптам, реально визначеним у `package.json`:

```bash
npm install       # встановлення залежностей
npm run dev       # локальний dev-сервер на localhost:4321
npm run build     # продакшн-збірка в ./dist/
npm run preview   # локальний перегляд зібраної версії
```
